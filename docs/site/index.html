Runtime overview

User input
  -> Ink UI (ui-ink)
  -> stdio request/event protocol
  -> Java stdio server (app-cli)
  -> prompt router / query engine (core)
  -> provider plugin (provider-*)
  -> tool runtime / permissions / hooks
  -> persisted session + state
  -> snapshot + events back to Ink
  -> normalized transcript render

Backend kernel

`core` owns the query engine, tool loop, session model, compaction, diagnostics, and provider routing.

Operator shell

`app-cli` owns command execution, stdio IPC, provider/session/model services, and Java entrypoints.

Terminal frontend

`ui-ink` owns prompt input, overlays, transcript normalization, render grouping, and status UX.

UI and transcript pipeline

The Ink app reconstructs renderable state from typed snapshots instead of rendering raw terminal text.

BackendSnapshot.messages
  -> reorderMessagesForUI(...)
  -> buildMessageLookups(...)
  -> groupToolUses(...)
  -> collapseReadSearchGroups(...)
  -> RenderableMessage[]
  -> Transcript / Message components

Important notes

  • The active runtime is the com.openclaude.* Gradle build, not the parallel io.openclaude.* trees.
  • The backend is manually composed in OpenClaudeApplication; there is no Spring container.
  • The UI is a separate process that talks to the backend over stdio, not an embedded Java terminal renderer.
  • Sessions, state, and session-memory are persisted under OPENCLAUDE_HOME or ~/.openclaude.
